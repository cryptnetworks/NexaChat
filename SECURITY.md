# Security policy

Do not open public issues for suspected vulnerabilities or include secrets, private content, or exploit details in ordinary reports. Use the repository's private vulnerability-reporting flow; if it is unavailable, contact the repository owner through a private channel listed on the hosting profile. Include affected versions, impact, reproduction steps, and suggested mitigations. Do not access other people's data or disrupt services while researching.

The security owner and repository-control process are documented in [repository controls](docs/security/repository-controls.md). The project never asks reporters to include credential values, private user content, complete logs, or exploit payloads in a report.

Only released versions designated as supported in release notes receive fixes.
The canonical [support and compatibility policy](docs/releases/support-compatibility.md)
defines security-maintenance objectives and end-of-support rules. The project
is pre-release and currently provides no supported production version.
