# Credential management

Authenticated accounts change their password with `POST /v1/account/password`. The request requires the exact trusted Origin, `x-nexa-csrf: 1`, and an active secure session cookie. It accepts only `currentPassword` and `newPassword`, each bounded to 12–128 characters before any Argon2 work. The current password is verified against the stored Argon2id hash; reuse of the current password is rejected. Wrong-current, reuse, stale-race, suspended-account, and unknown-account failures share the stable `authentication_failed` response.

A successful change uses credential-version compare-and-swap inside one PostgreSQL transaction. It replaces the hash, increments the credential version, revokes every prior session, appends the `account.credentials.change` audit event, and creates an immutable `credentials_changed` security notification. Exactly one concurrent writer can commit. After commit the server issues a new session at the new credential version and replaces the HttpOnly, SameSite=Strict cookie. A failure rolls back credentials, revocations, notification, and audit evidence together.

`POST /v1/auth/logout-all` similarly revokes every active session and records `account.sessions.revoke_all` plus `sessions_revoked`, then expires the caller cookie. Security notifications retain only an ID, account ID, allowlisted type, server correlation ID, timestamps, and fixed expiry. Audit events retain only allowlisted identity, target, action, outcome, reason, correlation, ordering, and integrity fields. Neither store accepts passwords, hashes, request bodies, cookies, tokens, addresses, or location fingerprints.

The web form uses password-manager autocomplete tokens, native bounds, a confirmation field, a keyboard-operable submit control, busy state, and a polite status region. Its generic error copy does not distinguish wrong-current, reuse, suspension, or a lost race. On success it clears all fields and reports that other signed-in devices were signed out.

Before release, run authentication unit and PostgreSQL HTTP integration tests, migration and backup/restore verification through the current schema, accessibility automation, rate-limit and audit tests, the sensitive-log scan, secret scan, production build/container checks, and the complete repository suite. During rollback, use only an application version that explicitly supports the deployed schema; otherwise fix forward or restore the complete verified pre-migration backup.
