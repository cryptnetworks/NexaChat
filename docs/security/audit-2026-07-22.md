## Review method

The review combined manual control-flow and data-flow inspection with negative
tests, concurrency tests, clean lock-based installation, dependency-tree and
registry reconciliation, audits, license and integrity policy checks, production
and OSV lock scans, builds, migration and recovery checks, performance and
connection-capacity tests, repository setting inspection, and workflow review.
Automated scanners were treated as evidence sources rather than substitutes for
authorization, failure-mode, privacy, and trust-boundary analysis.

Finding severity reflects exploit prerequisites and confidentiality, integrity,
and availability impact. `risk:high` on a tracking issue describes implementation
or regression risk and does not by itself mean the unresolved vulnerability is
high severity.

## Detailed findings

### SEC-01 — actor claims could bypass authentication in valid runtime composition

- **Severity / confidence:** High / High.
- **Classification:** CWE-306, OWASP A07 and A01.
- **Affected components:** community, space, and message creation handlers in
  `apps/server/src/app.ts`.
- **Prerequisite and impact:** a network client could reach a deployment
  composition with authentication configured but the optional permission gateway
  absent. An actor identifier in a request could be accepted without proving the
  matching session, allowing unauthorized creation under another identity.
- **Evidence and reproduction:** manual composition tracing found authentication
  selected conditionally on the permission gateway. Negative integration tests
  instantiate each supported composition and assert that an absent or mismatched
  session is rejected. Request details are intentionally omitted until the fix
  reaches the default branch.
- **Remediation:** authentication now depends only on the authentication runtime;
  optional authorization wiring cannot disable identity verification. Missing
  account rate limiting is also a startup/runtime error rather than a bypass.
- **Regression status:** covered in server authorization and message-lifecycle
  integration tests.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** recovery and identity verification remain roadmap work in
  #35 and #187-#189.

### SEC-02 — role data was not constrained to the evaluated community

- **Severity / confidence:** High / High.
- **Classification:** CWE-863, OWASP A01.
- **Affected components:** `packages/authorization/src/index.ts` and PostgreSQL
  authorization snapshots.
- **Prerequisite and impact:** an authenticated actor with role state in another
  community could receive a permission decision influenced by non-applicable role
  or assignment records, breaking tenant isolation.
- **Evidence and reproduction:** the evaluator previously selected assignments by
  actor without also requiring the evaluated community and a role applicable to
  that community. Decision-table tests use conflicting cross-community roles and
  require denial.
- **Remediation:** roles, assignments, and permission decisions are filtered by
  current community, role applicability, actor, and scope.
- **Regression status:** authorization unit and server integration suites cover
  cross-community assignment, global role, stale role, and denied scope cases.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** every database query must continue to include explicit tenant
  predicates; UI filtering is never authority.

### SEC-03 — concurrent message retries could duplicate effects or use stale state

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-362 and CWE-367, OWASP A04.
- **Affected components:** message command handling in `packages/domain`,
  `packages/postgres`, and `apps/server`.
- **Prerequisite and impact:** concurrent requests using the same idempotency key,
  or mutable space, permission, content-limit, reply, or slow-mode state changing
  before commit, could produce inconsistent HTTP and event outcomes.
- **Evidence and reproduction:** concurrency review found the first idempotency
  lookup and event/mention handling outside the authoritative serialized path.
  HTTP and PostgreSQL tests race identical and conflicting requests and mutate
  relevant state before commit.
- **Remediation:** the command uses a transaction advisory lock, deterministic
  fingerprint replay, mutable-state re-reads, transactional permission checks,
  one stable created-event identity, and mention/event processing only for the
  newly committed result.
- **Regression status:** domain, HTTP lifecycle, and PostgreSQL integration tests
  cover identical retry, conflicting retry, slow mode, archive, reply, limits,
  permission loss, event identity, and duplicate effects.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** the broader transaction-bound authorization contract remains
  open in SEC-13 / #203.

### SEC-04 — pending WebSocket admissions and sequence state were not fully bounded

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-400 and CWE-770, OWASP A04.
- **Affected components:** `apps/server/src/websocket.ts`.
- **Prerequisite and impact:** concurrent slow admissions could exceed configured
  connection caps, and high-cardinality space events could grow sequence state
  without a bound, increasing memory and connection pressure.
- **Evidence and reproduction:** admission counters previously covered established
  sockets only. Sequence state retained every observed space. Concurrency tests
  hold authentication open while attempting excess upgrades and drive bounded
  sequence eviction.
- **Remediation:** pending upgrades reserve global, canonical-address, and account
  capacity, expire after ten seconds, recheck drain/cap state, and are cleaned on
  shutdown. Sequence state uses a capacity derived from configured connections
  and subscriptions with inactive-first/LRU eviction.
- **Regression status:** WebSocket integration tests cover concurrent admission,
  shutdown overlap, capacity, cleanup, and sequence bounds.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** capacity must be requalified when connection or subscription
  defaults change.

### SEC-05 — local fixed-window limiters allowed unbounded identity buckets

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-400 and CWE-770, OWASP A04.
- **Affected components:** authentication, invitation, and discovery fallback
  limiters in `packages/auth` and `packages/domain`.
- **Prerequisite and impact:** high-cardinality anonymous identities could retain
  an unbounded number of in-memory counters, causing memory pressure when the
  distributed limiter is absent or degraded.
- **Evidence and reproduction:** the maps had no maximum cardinality. Unit tests
  fill each limiter, advance expiry, and assert fail-closed behavior and recovery.
- **Remediation:** configuration is validated, keys are deduplicated, expired
  entries are reclaimed, capacity is fixed, and a full active map denies new
  buckets.
- **Regression status:** authentication, invitation, and discovery tests cover
  capacity, expiry, invalid configuration, and duplicate keys.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** distributed Valkey limits remain the production control;
  local maps are a bounded degradation path, not a cross-instance quota.

### SEC-06 — browser mutations and navigation needed uniform trust checks

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-352 and CWE-601, OWASP A01 and A03.
- **Affected components:** web request helpers, desktop and web notification
  mutations, search, discovery, and link rendering in `apps/web` and
  `apps/server`.
- **Prerequisite and impact:** a browser-originated mutation without the standard
  CSRF marker could be rejected inconsistently, while an untrusted result URL
  could become an unsafe navigation target.
- **Evidence and reproduction:** manual call-site inventory found mutations that
  built headers independently and result components that rendered supplied
  destinations. Component and request tests supply cross-origin, scheme-relative,
  and script-like values.
- **Remediation:** all JSON mutations use one CSRF header helper; server checks
  exact trusted origin and marker. Search and discovery destinations must parse
  as same-origin path/query/fragment, while message links remain plain HTTP/HTTPS
  links with relationship protections and never trigger server preview fetching.
- **Regression status:** web HTTP, navigation, search, discovery, link, and
  notification tests cover rejection and accessible fallback text.
- **Fix:** commit `28d3d54` on `security/application-hardening`.
- **Residual risk:** browser-local drafts remain readable to a compromised browser
  profile, as documented in the threat model.

### SEC-07 — committed moderation restrictions were skipped with the production gateway

- **Severity / confidence:** High / High.
- **Classification:** CWE-863, OWASP A01.
- **Affected components:** `CommunityService.enforce` in `packages/domain`.
- **Prerequisite and impact:** a timed-out or banned member could retain message or
  real-time subscription access when the scoped production authorization gateway
  was active because committed restriction evaluation occurred only in the
  fallback branch.
- **Evidence and reproduction:** production wiring review showed an early return
  before the restriction lookup. A regression fixture uses a production-style
  gateway, commits a timeout, and requires both message-list and subscription
  authorization to deny.
- **Remediation:** current committed restrictions are evaluated before delegation
  to the permission gateway. Moderation commands remain separately authorized.
- **Regression status:** member-timeout tests cover HTTP-equivalent message access,
  WebSocket authorization, expiry, protected owner, bounded reason, and
  idempotency paths.
- **Fix:** commit `e767b0f` on `security/application-hardening`.
- **Residual risk:** multi-instance immediacy still depends on every path using the
  central service boundary and on transaction-bound revalidation in #203.

### SEC-08 — Valkey subscriber errors could terminate the process

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-248 and CWE-755, OWASP A04.
- **Affected components:** `packages/coordination/src/index.ts`.
- **Prerequisite and impact:** a subscriber connection error or disconnect could
  emit an unhandled event and terminate a server instance; an offline command
  queue could also create stale work during recovery.
- **Evidence and reproduction:** failure-injection tests drive duplicate-client
  errors, disconnect, reconnect, duplicate delivery, and cleanup.
- **Remediation:** publisher and subscriber disable offline queues, the subscriber
  owns a non-throwing degradation handler, and reconnect uses bounded exponential
  backoff with jitter and the configured circuit interval.
- **Regression status:** coordination tests cover error handling, retry bounds,
  duplicate identity, unsubscribe, and cleanup.
- **Fix:** commit `4008f16`, included in `main` through #198.
- **Residual risk:** fan-out can be temporarily degraded; durable HTTP state is the
  reconciliation source.

### SEC-09 — recoverable job completion used the wrong parameter position

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-682, OWASP A04.
- **Affected components:** `PostgresJobRecoveryStore` in `packages/postgres`.
- **Prerequisite and impact:** a leased background job completing without an
  error code could compare status against the wrong SQL parameter, leaving valid
  work incomplete and causing retries or stalled cleanup.
- **Evidence and reproduction:** query and parameter-list review found different
  shapes sharing one fixed placeholder. PostgreSQL integration tests complete and
  fail leased jobs with both shapes.
- **Remediation:** the expected-status placeholder and parameters are constructed
  deterministically for success and failure updates.
- **Regression status:** PostgreSQL job recovery and server integration tests
  cover success, failure, stale lease, version mismatch, retry, and restart.
- **Fix:** commit `6e2d45f`, included in `main` through #198.
- **Residual risk:** job schemas and every new update shape require integration
  coverage against PostgreSQL, not only an in-memory adapter.

### SEC-10 — recovery evidence checks could obscure the primary failure

- **Severity / confidence:** Low / High.
- **Classification:** protection-mechanism reliability; no specific CWE assigned.
- **Affected components:** backup restore fixture and artifact steps in
  `.github/workflows/backup-restore.yml`, `.github/workflows/verify.yml`, and
  `scripts/verify-backup-restore.sh`.
- **Prerequisite and impact:** schema evolution made the restore fixture omit new
  required message identity fields. If evidence or SBOM generation failed, an
  unconditional artifact upload could add a secondary failure and obscure the
  original result. The corruption control also overwrote a random ciphertext
  byte with a fixed value, so the controlled mutation could occasionally leave
  the component unchanged and report a spurious unsafe-success failure.
- **Evidence and reproduction:** the disposable restore was executed against all
  migrations; workflow paths were evaluated with missing evidence. A hosted run
  captured the no-op mutation case before any restore or retention claim was
  accepted.
- **Remediation:** restore data includes the stable idempotency fingerprint and
  event identity. Artifact uploads run only when their expected file exists while
  preserving the original job failure. The corruption fixture now appends a byte,
  deterministically changing both authenticated length and digest.
- **Regression status:** backup policy and disposable restore verification cover
  schema migration, content identity, evidence presence, deterministic corruption,
  and failure behavior.
- **Fix:** commits `25baead` and `037a90e`, included in `main` through
  #198.
- **Residual risk:** production restore and image checks still require a Docker
  environment and separately retained evidence.

### SEC-17 — synchronous stream completion could stall backup operations

- **Severity / confidence:** Low / High.
- **Classification:** CWE-362, OWASP A04.
- **Affected components:** encrypted component streaming in
  `scripts/backup/crypto.mjs` and retention pruning in
  `scripts/backup/command.mjs`.
- **Prerequisite and impact:** a writable destination could emit `finish`
  synchronously from `end()` before the backup operation registered its listener,
  or close without ever emitting `finish`. Either lifecycle could leave an
  unresolved operation and terminate with an unsettled top-level await,
  preventing verification, pruning, or restore evidence from completing. No
  backup content was accepted without its authentication and integrity checks.
- **Evidence and reproduction:** the disposable hosted restore reached the
  second verified backup during retention pruning and exited at the unresolved
  top-level await. A deterministic unit destination that emits `finish`
  synchronously reproduces the event ordering without external services.
- **Remediation:** both encryption and decryption establish completion listeners
  before calling `end()`, accept only successful `finish`, reject a premature
  close with stable `destination_closed_before_finish`, and remove all lifecycle
  listeners on settlement.
- **Regression status:** the focused encryption suite passes five tests,
  including synchronous completion in both directions and prompt rejection of
  premature closure. Hosted run 29966236758 completed the full disposable
  backup, corruption, missing-component, retention-prune, and isolated restore
  sequence in 231 seconds and retained its evidence artifact for 30 days.
- **Fix:** commit `a2118cc`, included in `main` through #198, and follow-up commit
  `3b64ac5` on `security/application-hardening`.
- **Residual risk:** future stream adapters must preserve bounded completion and
  cancellation behavior; production monitoring still owns provider-specific
  stalls beyond the verified disposable path.

### SEC-18 — fixed verification ports collided with runner sockets

- **Severity / confidence:** Low / High.
- **Classification:** protection-mechanism reliability, CWE-362.
- **Affected components:** clean-environment recovery in
  `scripts/verify-clean-environment.sh`.
- **Prerequisite and impact:** the harness derived PostgreSQL, Valkey, and object
  service host ports from its process identifier inside the operating system's
  ephemeral range. An unrelated active socket could occupy the selected port and
  abort clean-start, dependency-recovery, telemetry, or shutdown verification.
  The collision failed closed and did not allow a skipped check to pass.
- **Evidence and reproduction:** an exact hosted run reached the isolated Compose
  startup and failed when its selected Valkey port was already in use. The
  remaining checks and independent disposable restore retained their own results.
- **Remediation:** Docker first selects each service host port atomically from
  disjoint ranges outside the standard host ephemeral range. The harness then
  recreates the healthy disposable providers on those exact selected ports
  before constructing application endpoints, and asserts that every binding
  survives its stop/start recovery cycle.
- **Regression status:** shell syntax and security policy pass locally. Hosted
  run 29967515719 completed clean startup, all 45 migrations, required and
  optional dependency outages, stable binding recovery, telemetry, and graceful
  shutdown on branch head `00fcd1a900c0bb675f9dbc8c658b8fafa180180e`.
- **Fix:** commits `675455e`, `31a30d9`, `4a29548`, `da8e379`, and `2cbc272`
  on `security/ci-hardening`.
- **Residual risk:** the application listener remains in a bounded range below
  the standard Linux ephemeral range and still fails closed on a collision.

### SEC-19 — development providers were published on every host interface

- **Severity / confidence:** Medium / High.
- **Classification:** CWE-668, OWASP A05.
- **Affected components:** local provider publication in `docker-compose.yml`
  and development operations guidance.
- **Prerequisite and impact:** a developer running the default Compose stack on
  a reachable network exposed PostgreSQL, Valkey, and object storage on all host
  interfaces with documented local credentials. A network peer able to reach
  those ports could access development data, coordination state, or stored
  objects. Production Compose did not publish these providers.
- **Evidence and reproduction:** Compose short syntax omitted a host address for
  all three provider mappings. Docker documents that this binds published ports
  to all interfaces unless a host address is specified.
- **Remediation:** all development provider ports now default to `127.0.0.1` and
  require an explicit `NEXA_DEVELOPMENT_BIND_ADDRESS` override to widen access.
  The operations guide warns against widening it on an untrusted network.
- **Regression status:** security policy requires exactly three reviewed
  loopback-default provider bindings. Development and production Compose
  rendering and the clean-environment run passed in hosted run 29967515719.

## Authentication review

Local passwords use Node Argon2id with at least 19,456 KiB memory, two passes, one
lane, 16-byte salts, and 32-byte tags. Session tokens use 32 random bytes and only
their SHA-256 digests are stored. Comparison, session expiry, rotation,
credential-version invalidation, password change, account suspension/deletion,
logout, and bounded concurrent sessions are implemented with uniform public
errors and redacted security events.

Production cookies use the `__Host-` prefix, Secure, HTTP-only, SameSite=Strict,
path `/`, and bounded lifetime. Exact origin plus a mutation header protect the
cookie transport against CSRF. Canonical trusted-proxy parsing prevents arbitrary
forwarding headers from becoming rate-limit identity. Authentication, registration,
and other sensitive limiters fail closed when their bounded capacity or required
coordination is unavailable.

Password recovery, verification delivery, and MFA are not implemented and must
not be inferred from the login flow; #35 and #187-#189 track the reviewed design
and implementation.

## Authorization, API, and WebSocket review

The server now authenticates independently of optional authorization wiring.
Scoped permission evaluation filters roles and assignments to the current
community and resource hierarchy, enforces current membership/account state, and
uses sole-owner and protected-role checks. Timeouts and bans are evaluated for
both HTTP and WebSocket access even with the production permission gateway.

The API has 72 bounded schema-backed routes. Every non-health request consumes a
normalized route and canonical-address distributed limit; authenticated paths
consume an account limit, and expensive scopes add narrower limits. Bodies,
identifiers, content, metadata, pagination, filters, exports, reasons, retries,
and public errors are bounded. SQL uses parameters. Unsafe rich-link schemes and
unsafe internal navigation are not rendered, and the server does not fetch link
previews.

WebSocket upgrades enforce exact origin, current session, distributed admission,
pending and established connection caps, message and subscription limits,
heartbeat, token/permission revalidation, output backpressure, slow-consumer
closure, bounded sequence/deduplication state, and stable event identity. Valkey
fan-out preserves identity and handles duplicate delivery, subscriber failure,
reconnect, and degraded operation. HTTP remains the reconciliation source.

Issue #203 remains a release blocker because the repository still needs one
transaction-bound authorization contract for every sensitive mutation.

## Database, storage, privacy, and logging review

PostgreSQL migrations are forward-only, ordered, checksum verified, advisory-lock
serialized, and transactional. Queries are parameterized, pools and timeouts are
bounded, schema constraints cover stable identity and uniqueness, and recovery
jobs use leases and versions. Audit records and legal-hold directives have
append-only database enforcement and hash chains with exportable checkpoints.
Restore procedures preserve restricted/tombstoned state and replay cleanup.

Valkey keys are namespaced, bounded, expiring, and digest raw identities. Atomic
scripts prevent distributed limiter races. Valkey is never durable session,
authorization, message, unread, or audit truth. Subscriber reconnect is bounded
and the offline command queue is disabled.

The S3-compatible adapter validates endpoint policy, bucket/key shape, bytes,
timeouts, checksums, and provider errors. Buckets are private, credentials are
file-backed and bucket-scoped, and logs omit provider endpoints, keys, URLs,
digests, and content. Product attachments are not implemented.

Telemetry allowlists event names and label values and rejects identifiers,
addresses, credentials, content, paths, queries, URLs, and high-cardinality
attributes. Public errors omit stack, SQL, filesystem, host, token, and connection
details. Production nginx logs method, status, bytes, and duration only; metrics
are private. Push and desktop notification copy omit message text and private
community details. Browser drafts are local and account/space scoped but are not
encrypted against a compromised browser profile.
